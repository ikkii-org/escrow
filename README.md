# Ikki Escrow
An Anchor-based Solana smart contract handling escrow logic for head-to-head duels.

This project implements an on-chain escrow system where two players deposit SPL tokens (e.g., SKR) into a decentralized vault. A centralized authority (the game server) then settles the match, dynamically distributing funds to the winner and reserving a platform fee for the treasury.

## Codebase Architecture

The `escrow` directory is structured as a standard Solana Anchor workspace:

- `programs/ikki_escrow/`
  The Rust smart contract source code. This is where all on-chain logic resides.
- `sdk/`
  A TypeScript SDK providing helper functions. This wraps the raw Anchor RPC calls and handles deterministic Program Derived Address (PDA) derivations for the backend.
- `tests/`
  The Mocha/Chai integration test suite verifying the smart contract's state transitions.
- `target/`
  Compiled artifacts, IDL definitions (`target/idl/`), and TypeScript types generated by `anchor build`.
- `Anchor.toml`
  Configuration for the Anchor CLI, detailing the localnet program ID, network settings, and test scripts.

## Smart Contract Logic (`programs/ikki_escrow/src/lib.rs`)

The on-chain logic relies on strict state transitions enforced by Anchor's account constraints.

### 1. State Accounts

The program maintains two core data structures:

#### `PlatformConfig`
A singleton account that stores global platform settings.
- `authority`: The public key of the backend server allowed to settle matches.
- `fee_bps`: The platform fee taken from the total pot, represented in basis points (e.g., 250 = 2.5%).
- `treasury`: The public key where collected fees are sent.

#### `EscrowAccount`
A deterministic account (PDA) created for each individual duel. It tracks:
- `duel_id`: A 16-byte array representing the backend UUID.
- `player1` / `player2`: The public keys of the participants.
- `stake_amount`: The required token amount each player must deposit.
- `token_mint`: The specific SPL token accepted for this duel.
- `status`: An Enum (`Open`, `Active`, `Disputed`, `Settled`, `Cancelled`).
- `expiry`: A Unix timestamp dictates when an unfilled/unsettled duel can be reclaimed.

### 2. Instructions

The contract exposes 7 primary instructions to interact with the Escrow lifecycle:

- `initialize_platform` / `update_platform`
  Called by the admin to set or update the `PlatformConfig`.

- `create_escrow`
  Player 1 locks their stake. The program initializes the `EscrowAccount` PDA and creates an associated `TokenAccount` vault PDA. Player 1's tokens are transferred into the vault, and the status is set to `Open`.

- `join_escrow`
  Player 2 locks an identical stake. The tokens are added to the vault, and the status transitions from `Open` to `Active`.

- `settle_escrow`
  Can only be signed by the `PlatformConfig.authority`. The total pot is calculated, the fee is diverted to the treasury, and the remaining balance is sent to the winner. The status becomes `Settled`.

- `dispute_escrow` / `resolve_dispute`
  Provides a manual resolution path if a duel encounters a severe conflict. The status shifts from `Active` to `Disputed`, and then finally `Settled` via `resolve_dispute`.

- `cancel_escrow`
  Allows Player 1 to back out and reclaim their stake over a duel that is still `Open` (Player 2 has not joined yet).

- `claim_expired`
  A permissionless crank operation that checks the `expiry` timestamp. If the current network time exceeds the expiry and the duel is still `Open`, the contract refunds Player 1 and sets the status to `Cancelled`.

## TypeScript SDK (`sdk/src/sdk.ts`)

The SDK is designed to be imported directly into the TypeScript backend (`server/`). It abstracts away the complexity of building raw Solana transactions.

### PDA Derivations
Solana smart contracts do not have dynamic account mapping like Ethereum mappings. Instead, accounts are deterministically derived using seeds.
The SDK exports helpers to find these addresses:
- `findPlatformConfigPDA`
- `findEscrowPDA(duelId)`
- `findVaultPDA(duelId)`

### The `IkkiEscrowSDK` Class
Instantiated with an `AnchorProvider` and the generated `Program<IkkiEscrow>`. It provides easy-to-use asynchronous methods wrapping the smart contract instructions:
- `sdk.createEscrow(...)`
- `sdk.joinEscrow(...)`
- `sdk.settleEscrow(...)`

For user-facing actions (`createEscrow`, `joinEscrow`), the server will typically construct these instructions using the underlying Anchor methods and return them as an unsigned, serialized transaction buffer for the frontend wallet to sign. For authority-facing actions (`settleEscrow`), the server SDK signs the transaction directly using its stored private key.

## Tests (`tests/ikki_escrow.ts`)

The test suite simulates the complete end-to-end lifecycle on a local validator using `@coral-xyz/anchor` and `mocha`.

To run the test suite:
1. Ensure the Solana Local Validator is installed.
2. Ensure you have a local keypair at `~/.config/solana/id.json` (generate with `solana-keygen new` if missing).
3. Run `anchor test`.

The suite covers:
- Complete successful paths (Create -> Join -> Settle).
- Cancel and Expiry mechanisms.
- Dispute lifecycles.
- All authorization and invalid state transitions (e.g., preventing a user from settling their own match, preventing a join on a non-Open escrow).
